# Surviving Speakeasy Regen

Anything under `cli-sdk/internal/` or `cli-sdk/cmd/` is rewritten from
templates the next time `speakeasy run -t compass-cli` runs — and that
happens in CI on every spec change (and on `force_cli_regen=true`). The
released binary doesn't come from `mono` either; the workflow uploads
the regenerated `cli-sdk/` as an artifact, mirrors it into
`CompassLabs/cli`, and GoReleaser builds from there. So any hand-edit
to a `// Code generated by Speakeasy DO NOT EDIT.` file in mono is
**invisible to users** the moment the spec changes once.

Commit `31c3769d8` (table-unwrap + JSON indent) is the cautionary tale:
shipped, merged, then silently dropped by the next regen because it
relied on edits-to-generated-files with no re-application mechanism.

Three mechanisms below cover every kind of fix.

---

## (1) OpenAPI overlay — for anything spec-shaped

`/.speakeasy/cli-overlay.yaml` is the CLI-only OpenAPI overlay (the
python and typescript SDKs use the unoverlayed `compass-api` source
per `/.speakeasy/workflow.yaml`).

Use it for:

- **`operationId` rewrites** — Speakeasy turns each into a subcommand
  name. Already used to remove stutter (`v2_earn_vaults` → `vaults` so
  the CLI command is `compass earn vaults`, not `compass earn earn-vaults`).
- **Parameter/body `description` fixes** — when a generator-emitted
  help string is wrong or misleading.
- **`x-speakeasy-*` extensions** — `x-speakeasy-name-override`,
  `x-speakeasy-ignore`, `x-speakeasy-group`, etc.
- **`paths[...].remove: true`** — hide a route from the CLI entirely
  (the existing entries trim 82 v1 and v2/cctp paths down to the v2
  product surface).

The overlay is input to the generator, so changes here survive every
regen automatically. **Prefer this layer for anything that can be
expressed at the spec level.**

---

## (2) New files in unmanaged paths — for new packages and helpers

Speakeasy only touches files that begin with `// Code generated by
Speakeasy (https://speakeasy.com). DO NOT EDIT.` (and are referenced
in its `gen.lock`). Files without that header are inert during regen.

Two established examples in the tree:

- `cli-sdk/internal/cli/risk/` — an entire hand-written subcommand
  package. The workflow has a `sed` (lines 317–340 of
  `generate_combined_spec_and_sdks.yaml`) that re-adds its import and
  init call into the regenerated `internal/cli/root.go`. The package
  itself never gets touched.
- `cli-sdk/internal/output/overrides.go`,
  `cli-sdk/internal/output/overrides_test.go`,
  `cli-sdk/internal/flagutil/overrides.go` — helpers (and tests) used
  by the patches in (3). Without these files there'd be no place to
  put `normalizeForEncoding`, `AlreadyPrinted`, `targetsString`,
  `formatFastAPIDetail`, `errorStatusCodeFromHTTPMeta` — they'd have
  to live inside the generated files, blowing up patch size.

Use this layer for: **new packages, helper functions, helper types,
unit tests**. Anything that has its own file is here.

---

## (3) Post-regen patches — for edits inside generated files

For changes that *have* to live in a generated file (a call-site swap,
a new conditional inside a generator-owned function), keep the change
as a unified diff in `cli-sdk/scripts/post-regen/*.patch`. The
workflow re-applies them with plain `git apply` right after
`speakeasy run -t compass-cli`, before the artifact is uploaded.
(`--3way` is intentionally avoided: it requires the pre-image blob to
be in the object store, which CI's shallow checkout doesn't have, and
fails with a misleading "does not match index" error even when the
working-tree file matches byte-for-byte.)

### Layout

```
cli-sdk/scripts/
├── post-regen.sh                      # runner, applies all patches in lex order
└── post-regen/
    ├── 01-output.patch                # output/output.go fixes
    ├── 02-pretty.patch                # output/pretty.go fixes
    ├── 03-flagutil.patch              # flagutil/metadata.go fixes
    └── 04-main.patch                  # cmd/compass/main.go fixes
```

Naming convention: `NN-<package>.patch`. One patch per generated file
keeps blast radius small — if one fails to apply on the next regen,
only that file's fixes are affected.

### What each patch carries today

| Patch | Bugs fixed |
|---|---|
| `01-output.patch` | `bytes` import + `indentJSONIfValid` + raw-JSON-passthrough indent; `printTable` envelope unwrap + helpers (from `31c3769d8`); FastAPI `{"detail":[...]}` prettifier; YAML/TOON JSON round-trip; error-body fallback; error de-duplication via `AlreadyPrinted` wrap |
| `02-pretty.patch` | Pretty mode delegates list-shaped responses to `printTable` (from `31c3769d8`) |
| `03-flagutil.patch` | `--dry-run` no longer bypasses required-flag check; bare string flag values auto-quoted (so `--chain base` works) |
| `04-main.patch` | `cmd/compass/main.go` checks `output.IsAlreadyPrinted` before reprinting — completes the error-dup fix |

The helpers each patch depends on live in (2) — e.g. `01-output.patch`
calls `normalizeForEncoding` and `formatFastAPIDetail`, both defined
in `internal/output/overrides.go`.

### How the script behaves on failure

`cli-sdk/scripts/post-regen.sh` runs plain `git apply` per patch
(see the script's `--apply--` block for why `--3way` is deliberately
avoided). If any returns non-zero, the script exits non-zero and the
release fails. This is intentional — silent skipping is the regression
mode that bit `31c3769d8`. The recovery recipe is below in "When a
patch stops applying".

### Adding a new patch

```bash
# 1. Make the edit in the generated file in mono.
$EDITOR cli-sdk/internal/<file>.go

# 2. Verify the edit works (build, tests, smoke run).
( cd cli-sdk && go build ./... && go test ./... )
/Users/bekanay/.local/bin/compass <your-test-command>

# 3. Capture the diff against the last pristine baseline. The
#    cleanest source for "pristine" is the file's content immediately
#    before any hand-edits — typically the version at the parent of
#    the commit that introduced your edit, or the latest checked-in
#    output of speakeasy run.
git show <PRISTINE-REV>:cli-sdk/internal/<file>.go > /tmp/pristine.go

#    Then build a tiny git repo so the diff comes out in `git diff`
#    format (with the `diff --git a/X b/X` header that `git apply`
#    expects). The index OLD..NEW line in the output is ignored at
#    apply time — we use plain `git apply` for shallow-clone safety.
tmp=$(mktemp -d)
mkdir -p "$tmp/cli-sdk/internal" && cp /tmp/pristine.go "$tmp/cli-sdk/internal/<file>.go"
( cd "$tmp" && git init -q && git add -A && git commit -qm pristine \
  && cp /Users/bekanay/Projects/mono/cli-sdk/internal/<file>.go cli-sdk/internal/<file>.go \
  && git diff -- cli-sdk/internal/<file>.go ) \
  > cli-sdk/scripts/post-regen/NN-<package>.patch

# 4. Sanity-check the script still applies everything.
bash cli-sdk/scripts/post-regen.sh   # use a fresh worktree if possible

# 5. Update the "what each patch carries today" table above.
```

### When a patch stops applying

CI will fail with `git apply` rejecting the patch. Don't silence it.
The path forward leans on the `cli-sdk-pristine` artifact that the
workflow step right before `post-regen.sh` always uploads:

```bash
# 1. Grab the pristine from the failed run.
gh run download <RUN-ID> -n cli-sdk-pristine -D /tmp/pristine

# 2. Overwrite the failing file in your working tree with the new
#    pristine so you can see the template diff explicitly.
cp /tmp/pristine/internal/<file>.go cli-sdk/internal/<file>.go

# 3. Re-apply your intent by hand against the new pristine.

# 4. Regenerate the patch via the steps in "Adding a new patch" above.

# 5. Re-run post-regen.sh to confirm everything still applies.
bash cli-sdk/scripts/post-regen.sh
```

A failed patch means the Speakeasy template changed — the fix may
need to be rephrased to match the new surrounding code.

## Decision tree

```
Is the change expressible at the OpenAPI spec level
(naming, descriptions, hiding endpoints, extensions)?
  └── Yes → (1) Add to .speakeasy/cli-overlay.yaml

Is the change a new function/type/package, with no need to modify
existing generated code?
  └── Yes → (2) New file with no Speakeasy header

Does the change have to modify the body of a generated function?
  └── Yes → (3) Patch file under cli-sdk/scripts/post-regen/,
            plus helpers in (2) if the change has logic worth testing
```

The further left in this list a fix can live, the lower its
maintenance cost. Reach for (3) only when (1) and (2) won't do.

---

## Don'ts

- **Don't** hand-edit a `// Code generated by Speakeasy DO NOT EDIT.`
  file without a corresponding patch in `cli-sdk/scripts/post-regen/`.
  Commit `31c3769d8` did exactly that and the fix lived for 3 days.
- **Don't** sprinkle `// #region custom-fix-*` markers inside
  generated function bodies and assume Speakeasy preserves them.
  Speakeasy's region preservation only kicks in at template-emitted
  extension points (the `class-body-*` blocks in
  `internal/sdk/models/`). Markers anywhere else are decorative.
- **Don't** make `post-regen.sh` tolerant of partial failures. A
  half-applied patch set ships a half-broken binary.
- **Don't** put helpers inside a generated file just because the
  patch is "small." Put logic in (2) and keep the patch in (3) to a
  call-site swap. Smaller patches survive longer.

## Verifying after a regen

After any `speakeasy run -t compass-cli` (CI or local):

```bash
# 1. Patches must all be applied — script exits 0.
bash cli-sdk/scripts/post-regen.sh

# 2. Build and unit tests.
( cd cli-sdk && go build ./... && go test ./... )

# 3. Smoke battery for the priority fixes.
bin=$(go -C cli-sdk build -o /tmp/compass-rebuild ./cmd/compass && echo /tmp/compass-rebuild)
$bin earn earn-vaults --order-by tvl_usd --limit 1 -o toon \
  | grep -v omitzero >/dev/null   # TOON round-trip
$bin earn earn-vaults --order-by tvl_usd --limit 1 -o yaml \
  | grep -v "true: " >/dev/null   # YAML round-trip
$bin earn earn-vaults --order-by tvl_usd --chain base --limit 1 -q '.vaults|length' \
  | grep -q '^[0-9]'              # bare-string flag quoting
$bin earn earn-positions-all --owner notanaddress 2>&1 \
  | grep -q "Validation error"    # FastAPI prettifier + single render
$bin earn earn-create-account --dry-run 2>&1 \
  | grep -q "missing required"    # dry-run flag check
$bin earn earn-vaults --order-by tvl_usd --limit 2 -o table 2>&1 \
  | grep -q "VAULT_ADDRESS"       # list-envelope unwrap (31c3769d8 fix)
```

If any step fails, the corresponding patch needs revisiting.
